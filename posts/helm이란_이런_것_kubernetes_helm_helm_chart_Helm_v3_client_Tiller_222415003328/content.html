helm 이란

kubernetes helm 이란?

Docker가 나오면서 container 혁신이 이루어지고,

container를 쉽게 관리/배포해주는 kubernetes가 나오게 되었다.

(일종의 container orchestration)

​

이 kubernetes를 package로 해서 관리해주는 것이 바로 helm이다.

apt나 yum과 같이 필요한 패키지를 쿠버네티스 위에 편리하게 설치할 수 있게 도와주는 툴, 즉 쿠버네티스 패키지 매니저 이다. 복잡한 어플리케이션도 helm chart 하나면 뚝딱 쿠버네티스 위에 설치할 수 있다.🧞‍♂️

​

쿠버네티스 자원들을 묶어 하나의 패키지로 배포/다운로드 할 수 있게하고

쿠버네티스 클러스터 위에 자신이 배포한 서비스를 Helm 통해 패키지화하여 배포하면

다른 사용자들이 Helm을 통해 그들의 쿠버네티스 클러스터에 패키지를 다운받아 사용할 수 있다.

​

Helm은 이러한 패키지를 'Helm Chart'라는 포맷으로 정의하여 관리한다.

Helm Chart는 하나의 패키지를 설치하기 위한 파일들의 구성을 정의하며,

쿠버네티스 자원들의 yaml 파일들을 포함한다.

​

Helm chart 📋

Helm Chart 형식(아래 그림의 Chart.yml)으로 쿠버네티스 패키지들이 배포/다운로드 된다.

​

​

​

Helm is a tool for managing Kubernetes charts. Charts are packages of pre-configured Kubernetes resources.

​

헬름은 쿠버네티스 차트를 관리하기 위한 도구,

차트는 사전 구성된 쿠버네티스 리소스의 패키지다.

​

​

helm chart에는 크게 빼대를 이루는 templates와,

그 빼대에 살을 채우는 values.yaml 파일이 있다.

​

values.yaml파일에는 각 인프라 설정에 따라서 세부적으로 customization하는 부분이다.

이 두개의 조합으로 하나의 패키지가 완성되어 쿠버네티스 위에 올라간다.

​

Helm은 chart라고 부르는 package format을 사용하는데

chart는 쿠버네티스 리소스를 describe 하는 파일들의 집합이다.

​

헬름은 패키지 관리 도구고, 차트가 리소스를 하나로 묶은 패키지에 해당한다.

헬름으로 차트를 관리하는 목적은 자칫 번잡해지기 쉬운 매니페스트 파일을 관리하기 쉽게 하기 위한 것이다.

​

실무에서는 로컬 및 운영 클러스터를 막론하고 여러 환경에 배포해야 하는 애플리케이션은 모두 차트로 패키징해 kubectl 대신 헬름으로 배포 및 업데이트를 수행한다. 그 대신 이미 배포된 리소스에 대해 kubectl로 수정한다.

​

​

특정 directory 하위에 모여있는 파일들을 통틀어서 chart라고 부르는데..

쉽게 예를들어 WordPress chart는 다음과 같은 구조를 가진다.

wordpress/ derectory 밑으로 :

wordpress/ Chart.yaml # A YAML file containing information about the chart LICENSE # OPTIONAL: A plain text file containing the license for the chart README.md # OPTIONAL: A human-readable README file requirements.yaml # OPTIONAL: A YAML file listing dependencies for the chart values.yaml # The default configuration values for this chart charts/ # A directory containing any charts upon which this chart depends. templates/ # A directory of templates that, when combined with values, # will generate valid Kubernetes manifest files. templates/NOTES.txt # OPTIONAL: A plain text file containing short usage notes 출처 : https://docs.helm.sh/developing_charts/#charts

위와 같은 구조로 되어 있고 version 정보가 붙어서 tgz 형태로 압축 패키징 된다.

1.2.3 version의 nginx chart는 nginx-1.2.3.tgz 이런식으로.

​

​

Repository 🗂

저장소

차트 저장소로, 차트를 모아두고 공유하는 장소.

​

local

헬름 클라이언트가 설치된 로컬 리포지토리로, 로컬에서 생성한 패키지가 존재한다.

stable

안정 버전에 이른 차트가 존재하는 리포지토리다. 안정된 보안 수준과 기본 설정값을 포함하는 등 일정한 요건을 만족하는 차트만 제공될 수 있다.

incubator

요건을 만족하지 못하는 차트가 제공되는 리포지토리다. stable로 넘어갈 예정인 차트가 제공된다.

​

stable 리포지토리는 기본값으로 사용되는 리포지토리로, 깃헙 helm/charts 리포지토리에 저장된 차트를 사용할 수 있다. incubator 리포지토리는 기본값으로 사용되지는 않지만 다음과 같이 리포지토리를 추가할 수 있다.

helm repo add incubator https://kubernetes-charts-incubator.storage.googleapis.com/

​

리포지토리에서 다음과 같이 차트를 검색할 수 있다.

helm search 명령에 지정한 키워드로 검색도 가능하다.

helm search

​

Release

릴리즈

k8s cluster에서 구동되는 차트 인스턴스.

일반적으로 동일한 Chart를 여러 번 설치할 수 있고, 이는 새로운 Release로 관리된다.

릴리즈 될 때 패키지된 차트와 Config가 결합되어 정상 실행된다.

​

아래 순서와 같이 연계된다.

k8s cluster 내부에 Helm Chart를 ▶ 각 설치에 따른 원하는 Repository에서 검색 후 설치 새로운 Release 생성

​

​

Helm v3 client, Tiller

Helm은 원래 크게 client와 server(tiller). 이렇게 두가지 파트로 나뉘어져 있었다.

client는 end user를 위한 command line client이고, 주로 local chart 개발이나 repository managing, server(tiler)에 chart 설치 요청등 주로 chart를 관리하는 파트이고

​

Tiller 라고도 부르는 server는 in-cluster 서버로 chart의 배포, 릴리즈를 관리하는 파트이다.

​

그렇지만 Helm v2에서 v3로 변하면서 아키텍처적와 개념이 많이 바뀌었다.

주요 내용은 'Helm Server 부분이였던 tiller 부분이 삭제되었다'라는 점이고,

더 자세한 사항은 아래 공식 사이트 링크를 참고!

​

https://helm.sh/docs/topics/v2_v3_migration/

Migrating Helm v2 to v3

Helm 사용 : macOS

helm 설치

brew install helm

​

쿠버네티스 클러스터가 준비되었다면

다음의 명령어로 설치 스크립트를 다운 받는다.

curl https://raw.githubusercontent.com/helm/helm/master/scripts/get > get_helm.sh chmod 700 get_helm.sh ./get_helm.sh

​

아래 명령어로 해도 좋다.

helm repo add incubator https://charts.helm.sh/incubator

설치가 완료.

​

tiller 서비스 account를 생성, helm 초기화하고 cluster-admin role을 부여.

kubectl -n kube-system create sa tiller
kubectl create clusterrolebinding tiller --clusterrole cluster-admin --serviceaccount=kube-system:tiller

​

tiller 설치

helm init --service-account tiller
