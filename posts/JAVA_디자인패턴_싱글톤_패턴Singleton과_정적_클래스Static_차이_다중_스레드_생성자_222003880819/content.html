0. 싱글톤 패턴(Singleton) 이란?

최소 한번만 메모리를 할당하고

그 메모리에 인스턴스를 만들어 사용하는 디자인 패턴.

단 하나의 인스턴스를 생성해 사용하는 디자인 패턴이다.

​

인스턴스가 오직 하나만 생성되는 것을 보장하고 어디서든 이 인스턴스에 접근 가능한 패턴이다.

인스턴스가 필요할 때 똑같은 인스턴스를 만들어 내는 것이 아니라

기존의 동일한 인스턴스를 사용하게 한다.

다시말해 하나의 인터페이스만 만드는 책임이 있고,

인스턴스를 반환하는 행동을 통해 모든 클라리언트에게 동일한 작업을 보내는 것.

​

​

생성자가 여러차례 호출되더라도 실제로 생성되는 객체는 하나고,

최초 생성 이후에 호출된 생성자는 최초에 생성한 객체를 반환한다.

​

​

1. 싱글톤 패턴을 사용하는 이유

고정된 메모리 영역을 얻으면서 한번의 new로 인스턴스를 사용하기 때문에 메모리 낭비가 없다.

또 싱글톤으로 만들어진 클래스의 인스턴스는 전역 인스턴스기 때문에 다른 클래스의 인스턴스들이 데이터를 공유하기 쉽다.

​

따라서 공통된 객체를 여러개 생성해서 쓰는 상황에서 많이 사용한다!

예를 들어 작은 회사에서 쓰는 한 대의 프린터기, 대화상자나 로그 기록 객체 등.

인스턴스가 절대적으로 한개만 존재하는 것을 보증하고 싶을 경우 사용한다고 볼 수 있다.

​

2. 싱글톤 패턴의 문제점

싱글톤 인스턴스가 너무 많은 일을 하거나

많은 데이터를 공유시킬 경우에

다른 클래스 인스턴스 간에 결합도가 높아져 객체지향 설계 원칙에 어긋나는'개방-폐쇄' 원칙을 위배하게 된다.

따라서 수정이나 테스트하기가 어려워진다.

또 멀티쓰레드 환경에서 동기화 처리를 안하면 인스턴스가 두개 생성된다는 문제가 발생하기도 한다.

​

공유되는 데이터는 그렇지 않은 상황.

방법 2.

인스턴스를 만드는 메서드에 동기화 하는 방법.

3-e. class Printer { private static Printer printer = null; private int count = 0; private Printer() { } public static synchronized Printer getPrinter() { if (printer == null) { try { Thread.sleep(1); printer = new Printer(); } catch (InterruptedException e) { } } return printer; }

프린터 클래스의 객체를 얻을 수 있는 get Printer 메서드를 동기화 시키는 것이다.

순차적으로 접근하게끔!

여기에서는 해당 메서드에 synchronized를 추가해서,

다중 쓰레드로 접근하는 것은 한번에 하나 씩, 비동기 방식으로 한다.

'지금 누가 접근하고 있으니, 잠시 기다려' (이렇게 하나하나 기다리게)

​

그래서 이 synchronized는

다중 스레드 환경에서 동시에 여러개가 메소드에 접근하는 것을 방지하고

스케쥴링 해주는 것 이라 생각하면 되겠다.

결과를 살펴보면,

비동기로 접근을 하지만 실제로 만들어진 프린터 객체는 다 동일하다.(id값 동일)

한번만 생성이 된 것이다.

​

마지막 결과 값도 보면..

최초 생성 이후 호출된 생성자는 최초에 생긴 객체를 반환.

장점

단점 / 문제점

고정된 메모리 영역을 얻으면서

한번의new로 인스턴스를 사용하기 때문에 메모리 낭비를 줄일 수 있다.

싱글톤 인스턴스가 너무 많은 일을 하거나

많은 데이터를 공유시킬 경우

다른 클래스의 인스턴스들 간에 결합도가 높아져 '개방 폐쇄 원칙'에 위배된다. 수정과 테스트가 어려워짐.

싱글톤 클래스의 인스턴스는 전역 인스턴스이기 때문에 다른 클래스의 인스턴스들이 데이터를 공유하기 쉽다.

동기화 처리(synchronized)를 하지 않으면 인스턴스가 두개가 생성되는 일이 생길수 있다.(주의)

공통된 객체를 여러개 생성해 사용할 때 좋다.

(쓰레드풀, 캐시, 로그 기록 객체 등)

인스턴스가 절대적으로 한개만 존재하는 것을 보증하고 싶을 때 사용

5. 싱글톤 패턴과 Static(정적 클래스)

드디어

그런데 정적 클래스를 사용해도 싱글톤 패턴과 동일한 효과를 얻을 수 있다!?

아래는 싱글톤을 사용하지 않고 정적 메소드로만 구현한 정적 클래스이다.

class Printer0504 { private static int count = 0; private Printer0504() { } public synchronized static void print(String str) { count++; System.out.println(str + count); } } class User extends Thread { private String name; public User(String name) { this.name = name; } public void run() { Printer0504.print(this.name + "print using" + "."); } } public class Client0504 { private static final int User_NUM = 5; public static void main(String[] args) { User[] user = new User[User_NUM]; for (int i = 0; i < User_NUM; i++) { user[i] = new User((i + 1) + "-user"); user[i].start(); } } }

​

출력해보면 이렇게 싱글톤과 동일한 효과를 얻을 수 있다는 것을 알 수 있다.

오히려 싱글톤 패턴에서 런타임 시에 바인딩이 되는 인스턴스 보다,

정적 메서드가(static) 성능이 더 우수하기도 하다.

그렇다면 정적 메서드가 더 좋은걸까? 싶기도 하지만,

​

이 정적메소드를 사용할 수 없는 경우가 있다.

바로 Interface를 사용할 때.

Interface는 static 을 지원하지 않기 때문에, 싱글톤을 사용해 구현 해야한다.

정적메소드(static)과 싱글톤(Singleton)의 차이에 대해 조금 더 자세히 알아보자.
