1. IT 업계의 트렌드, MSA

넷플릭스가 서버를 1,000조각으로 쪼갠 이유, 왜 MSA?

​

암스트롱.

해당 프로젝트를 선정한 이유와 관련이 있는 단어이다.

우주 탐사 얘기의 암스통이 아니라,

‘애플리케이션 현대화(AM)’ 프로젝트를 강하게(strong) 추진하자는 신조어다.

어플리케이션 현대화(Application modernization)는 기존 레거시 시스템을 현대 컴퓨터 프로그래밍 언어, 소프트웨어 라이브러리, 프로토콜 또는 하드웨어 플랫폼으로 변환 또는 이식하는 것을 말한다.

​

​

이 애플리케이션 현대화를 위한 디자인 패턴은 아래의 표와 같이 3가지가 있고,

해당 프로젝트에서는 마이크로서비스 아키텍처를 활용했다.

​

이 ‘암스트롱’의 핵심은 기존 IT 시스템을 수천 수백 개로 쪼.개.는. 것이다.

​

기존 시스템은 간단한 업데이트를 위해 서버 전체를 멈춰야 하는 경우가 적지 않은 상황.

마치 손가락을 다쳐도 일단 전신 마취를 해야하는 상황과 같은데,

애플리케이션 현대화는 이러한 상황에서 부분마취를 할 수 있게 해준다고 할 수 있다.

복잡한 시스템을 여러 개의 자율적인 조직으로 분화하여, 각종 업데이트 요구에 빠르게 부분마취로 대응할 수 있게 되는 것이다.

​

​

그래서 이 IT 시스템을 수천 수백 개로 쪼개는 기술 바로,

​

​

2. 'MSA' 란?

' Microservices Architecture '

​

​마이크로 서비스 아키텍처는, 아주 작은 단위로 동작하는 서비스가 구동되도록 시스템 및 소프트웨어의 구성과 구성요소 간의 관계를 정의한 아키텍처이다.

​

‘쪼개야 산다’

​

하나의 애플리케이션을 구성함에 있어 분할된 다수의 서버 또는 컨테이너를 통해 애플리케이션 기능뿐만 아니라 데이터까지 분리하여 격리된 독립된 환경으로 구성되는 점이 특징이라 할 수 있겠다.

​

이 좋다는 MSA 아키텍처는 우리가 잘 알고 있는 넷플릭스, 우버 택시가 이를 도입해 아주 대박을 쳤는데

적용 사례를 한번 알아보도록한다.

​

3. MSA 적용사례, Uber(우버) 🚖

MSA 아키텍처 VS 모노리틱 아키텍쳐(Monolithic Architecture)

'모놀리식' 이라는 아키텍처와 함께,

첫번째로 우버의 적용 사례에 대해 알아보자.

​

마이크로 서비스 아키텍쳐를 이해하려면 먼저 모노리틱 아키텍쳐 스타일에 대해서 이해해야 한다.

모노리틱 아키텍쳐 스타일은, 기존의 전통적인 웹 시스템 개발 스타일로

하나의 애플리케이션 내에 모든 로직들이 모두 들어 가 있는 '통짜 구조' 이다.

​

​

​

​

한 온라인 쇼핑몰 어플리케이션이 있을 때, 톰캣 서버의 WAR 파일(웹 애플리케이션 패키징 파일)내에 사용자 관리, 상품, 주문 관리 모든 컴포넌트들이 들어 있고, 이를 처리하는 UX 로직까지 하나로 포장돼서 들어가 있는 것이 이 모노리틱 구조.

​

모노리틱 아키텍쳐도 처음엔 나쁘지 않다고 생각했다.

이 방식은 개발되어 있는 다 환경이 같으니 일단 테스트나, 같은 서버를 하나 더 만드는 등의 작업이 복잡하지 않다는 장점이 있긴하니..

그래서, 당시 우버도 모노리틱 아키텍처로 만들어 졌었다.

https://brunch.co.kr/@yesjun/2

​

그러나 우버는 곧 글로벌 서비스를 시작했고, 확장성과 지속적인 연동 문제에 직면하게 되었다.

다음은 당시 우버가 직면했던 몇 가지 중요한 난제이다.

​

우버가 직면한 문제점에서 볼 수 있듯, 모놀리틱 구조의 장,단점은 다음과 같다.

모노리틱 구조의 장점

모노리틱 구조의 단점

1. 어떤 기능(서비스)이든지 개발되어있는 환경이 같아서 복잡하지않음.

​

2. 쉽게 고가용성 서버 환경을 만들 수 있다.

( 같은 어플리케이션으로 하나더 만들면 됨)

​

3. End-to-End 테스트가 용이함. (MSA의 경우 테스트에 필요한 서비스들을 모두 동작시켜야함)

1. 한 프로젝트의 덩치가 너무 커져서 어플리케이션 구동시간이 늘어나고 빌드,배포 시간도 길어진다.

​

2. 조그마한 수정사항이 있어도 전체를 다시 빌드하고 배포를 해야한다.

​

3. 많은 양의 코드가 몰려있어 개발자가 모두를 이해 할 수 없고 유지보수도 힘들다.

​

4. 일부분의 오류가 전체에 영향을 미친다.

​

5. 기능별로 알맞는 기술, 언어, 프레임워크를 선택하기가 까다롭다.

우버에서도 이 같은 이유로,

또 직면한 문제를 해결하고자,

우버는 기존 모노리틱 아키텍처에서 마이크로서비스 아키텍처로 이전하게 되었다.

도입 후, 가장 큰 변화는 운전자와 승객을 연결하는 API 게이트웨이의 등장이다.

​

API 게이트웨이로 부터 각 기능들의 모든 내부 요청이 연결되어 있고,

또 각각의 서비스는 독립적으로 분리되어 배포 가능하고 독립적으로 기능을 수행하게 되었다.

이렇게되면 요금청구 서비스에 변화가 있을 때, 딱 요금청구 서비스만 수정하여 배포하고, 다른 서비스는 배포할 필요가 없어진다! 즉, 각각의 기능을 독립적으로 적용 및 확장할 수 있게 된 것이다.

​

여기서 MSA 구조의 장점을 짚고 넘어가자.

MSA 아키텍처의 장점

단점

1. 기능별로 마이크로서비스를 개발하고, 작업 할당을 서비스 단위로 하면 개발자가 해당 부분을 온전히 이해할 수 있다.

​

2. 새로 추가되거나 수정사항이 있는 마이크로서비스만 빠르게 빌드, 배포가 가능하다.

​

3. 해당 기능에 맞는 기술, 언어 등을 선택하여 사용할 수 있다.

​

4. 일부분의 오류가 있으면 해당 기능에만 오류가 발생하고 그 부분만 빠르게 고쳐서 정상화가 가능하다.

?

단점은 가장 마지막에 알려드리겠습니다ㅎㅎ🤓

3. MSA 적용사례, 넷플릭스 🍿

넷플릭스 OSS

MSA 도입 사례 중 가장 유명한 기업은 사실~~

넷플릭스다.

​

Netflix는 기존 Legacy 시스템은 MSA로 모두 전환하여 운영/개발 상의 효율성을 극대화 하였다.

https://about.netflix.com/ko/news/completing-the-netflix-cloud-migration

​

2007년 심각한 데이터베이스 손상으로 3일간 서비스 장애를 겪은 넷플릭스는, 신뢰성 높고 수평 확장이 가능한 클라우드 시스템으로 이전할 필요성을 느꼈다. 단순히 플랫폼 이전만으로는 기존의 문제점과 한계를 탈피할 수 없다고 판단한 넷플릭스는 고가용성, 유연한 스케일링, 빠르고 쉬운 배포를 위해 MSA를 선택 했다고 한다. 그리고 MSA 전환을 위한 기술들을 도입하여 무려 7년에 걸쳐 클라우드 환경으로 이전했다.

​

🍿Netflix OSS

(Open Source Software)

이 거대한 이전 과정에서 넷플릭스가 경험한 노하우와 문제해결 방법을 공유하기 위해 MSA 전환 기술을 오픈소스로 공개 하였다!👏🏻

이렇게 탄생한 넷플릭스 OSS는 MSA를 도입하려는 많은 사람에게 좋은 선택지가 되고 있다.

​

4. 프로젝트, Spring Cloud Netflix

프로젝트에 MSA를 도입하려는 나같은 사람에게도 큰 도움이 되었다. ^_^

​

먼저 클라우드 환경에 특화된 프레임워크인 Spring Cloud를 살펴보자.

Spring Cloud는, Spring boot를 기반으로 MSA 구축에 특화된 라이브러리들의 집합이다.

​

​

​

Spring Cloud에는 Eureka, Hystrix, Ribbon, Zuul 등 많은 넷플릭스 OSS가 통합되어 있다.

여기서 알아볼 내용은 넷플릭스 오픈소스 중에서도 Spring Cloud에 속해 있는 Spring Cloud Netflix가 되겠다.

진행할 프로젝트에서는 이 Spring Cloud Netflix에 속해 있는 히스트릭스, 터빈, 유레카, 컨피그 서버, 주울 서비스를 사용할 예정이다.

4. 프로젝트,

MSA, 정말 필요한가?

MSA 아키텍처의 장점

SA 아키텍처의 단점

1. 기능별로 마이크로서비스를 개발하고, 작업 할당을 서비스 단위로 하면 개발자가 해당 부분을 온전히 이해할 수 있다.

​

2. 새로 추가되거나 수정사항이 있는 마이크로서비스만 빠르게 빌드, 배포가 가능하다.

​

3. 해당 기능에 맞는 기술, 언어 등을 선택하여 사용할 수 있다.

​

4. 일부분의 오류가 있으면 해당 기능에만 오류가 발생하고 그 부분만 빠르게 고쳐서 정상화가 가능하다.

1. 무엇보다 관리가 힘들다. 작은 여러 서비스들이 분산되어있기 때문에 모니터링이 힘들다.

​

2. 서로를 호출하여 전체 서비스가 이루어지기 때문에 무조건 다른 서비스를 호출하는 코드가 추가되는데

​

이부분이 모놀리식 아키텍쳐의 개발보다 조금 까다롭다.

​

3. 통신관련 오류가 잦을 수 있다. 마이크로 서비스들 끼리 계속 서로 통신을 하다보니 모놀리식 아키텍쳐에 비해 통신관련 오류가 잦았다.

​

4. 테스트가 불편하다. 예로 End-to-End 테스트를 위해 UI, Gateway 등등 여러개의 마이크로 서비스를 구동시켜야 했었다.
