2. JPA 사용

JPA에 대한 이론은 김영한님의 자바 orm 표준 jpa 프로그래밍으로 학습했고

실습은 구멍가게코딩단의 코드로 배우는 스프링부트로 학습했다.

그렇게 이번 프로젝트에서 데뷔한 JPA.

JPA는 ORM(Object-Relational Mapping) 기술이다.

'ORM (Object Relational Mapping)'

RDB 테이블을 객체지향적으로 사용하기 위한 기술. 말그대로 객체 관계 매핑

ORM이란 이런 것.

​

객체는 객체대로 설계하고, 관계형 데이터베이스는 관계형 데이터베이스대로 설계해서 ORM 프레임워크가 중간에서 매핑해준다. jpa를 사용해서 jdbc를 사용하는 것이다.

JPA는 어플리케이션과 DBMS 사이의 인터페이스 역할을 해주기 때문에 개발자는 JPA 인터페이스에 맞춰서 구현되어 있는 기능을 사용하면 된다. 다음은 JPA가 동작하는 과정이다.

​

JPA를 사용하기로 결심한 이유는 역시 위에서 언급한 문제점들을 해결하기 위해서이다.

​

JPA 사용 장점

- 패러다임의 불일치를 해결한다. 객체지향적으로 데이터를 관리할 수 있기 때문에 비즈니스 로직에 집중 할 수 있으며, 객체지향 개발이 가능하다.

- 테이블 생성, 변경, 관리가 쉽다. (JPA를 잘 이해하고 있는 경우..)

- 로직을 쿼리에 집중하기 보다는 객체자체에 집중 할 수 있다.

- 빠른 개발 가능

​

단점

- 어렵다. 장점을 더 극대화 하기 위해서 알아야 할게 많다.

- 잘 이해하고 사용하지 않으면 데이터 손실이 있을 수 있다. (persistence context)

​

3. Spring Data JPA

+ 쿼리 메소드

JPA를 공부하다보니, 책의 저자인 김영한님께서 이 jpa를 더 편하게 쓸 수 있는 기술에 대해 알려주셨다.

바로 Spring Data JPA 다.

​

Spring Data JPA는 Spring에서 제공하는 모듈 중 하나로, 개발자가 JPA를 더 쉽고 편하게 사용할 수 있도록 도와준다. JPA를 한 단계 추상화시킨 Repository라는 인터페이스가에 정해진 규칙대로 메소드를 입력하면, Spring이 알아서 해당 메소드 이름에 적합한 쿼리를 날리는 구현체를 만들어 Bean으로 등록해준다.

4. JPQL

JPA, springDataJPA는 다양한 쿼리 방법을 지원한다. JPQL, JPA Criteria, QueryDSL 등..

그 중 가장 기본맛인 JPQL에 대해 알아보자.

​

JPQL은 Java Persistence Query Language.

객체지향 쿼리 언어다.

Member member = em.find(Member.class, 1);

SQL처럼 테이블 대상이 쿼리가 아니라 엔티티 객체를 대상으로 쿼리한다.

예를들어, 테이블이 아닌 객체를 대상으로 검색하는 객체지향 쿼리라고 이해하면 될 것 같다.

JPA에서 단순히 PK값으로 find()만 하는것이 아니기 때문에 여러가지 조건으로 검색하는 복잡한 쿼리를 위해 JPQL이 존재한다.

JPQL을 분석하고, JPA가 적절한 SQL을 만들어 데이터베이스를 조회하는 식이다.

​

> member entity

@Entity(name = "Member") public class Member {      @Column(name = "name")     private String username;     // ... }

> JPQL

String jpql = "select m from Member as m where m.username = 'kim'"; List<Member> resultList = em.createQuery(jpql, Member.class).getResultList();

​

이렇게 메소드 이름을 분석해서 JPQL 쿼리를 실행하기도 하고, @Query 어노테이션을 사용해서 직접 JPQL을 지정할 수도 있다.

public interface MemberRepository extends JpaRepository<User, Long> { @Query("select m from Member u where m.name = ?1") List<User> findByName(String name, Pageable pageable); }

​

5. QueryDSL

Java 코드로 JPQL을 작성할 수 있다.

JPQL로도 해결하기 어려운 동적쿼리나 더욱 복잡한 쿼리를 해결하기 위해 QueryDSL이 필요하다.

복잡한 검색 조건이나 join 등의 기능도 구현이 가능하다.

​

JPA를 사용한다고 가정해보자. 간단한 쿼리라면 인터페이스에 메서드 명세만 잘 정의해 주면 별다른 문제 없이 사용할 수 있다. 예를 들면 아래처럼 “제목에 특정 문자열이 포함된 기사를 조회”하는 메서드처럼 말이다.

Article findByTitleContains(String title);

​

조금 더 복잡한 쿼리가 필요한 경우에는 어떨까? 앞서 살펴본 것처럼 단순히 특정 문자열이 제목에 포함된 기사를 조회하는 것이 아니라 기사를 작성한 사용자의 레벨을 기준으로 조회한다고 가정해보자.

​

이런 경우 JPA 자체 제공 메서드만으로 해결하기 어렵다. 네이티브 쿼리로 해보자.

@Query(value = "SELECT id, title, user_id FROM article WHERE user_id IN (SELECT id FROM user WHERE level > :level)", nativeQuery = true) List<Article> findByLevel(String level);

흠.. 가독성도 별로고 문자열을 이어 붙여가며 직접 작성하기 때문에 오타가 발생하기 딱 좋다.

​

이 코드를 Querydsl로 변경하면 어떻게 될까?

public List<Article> findByUserLevel(String level) { QArticle article = QArticle.article; QUser user = QUser.user; return queryFactory.selectFrom(article) .where( article.userId.in( JPAExpressions .select(user.id) .from(user) .where(user.level.gt(level)) ) ) .fetch(); }

언뜻보기엔 코드양만 늘었을지 몰라도,

개발 환경을 고려보면.. 가독성도 좋은데 친절하게 컴파일 오류까지 발생시켜준다.

이번 프로젝트에서 사용해보니

Querydsl의 최대 장점은 '검색조건에 따라 동작하는 동적쿼리를 생성하기 편하다.' 인 것 같다.

​

*QueryDSL은 JPQL을 코드로 작성할 수 있도록 도와주는 빌더 역할이기 때문에 결국 jpql로 변환된다.
