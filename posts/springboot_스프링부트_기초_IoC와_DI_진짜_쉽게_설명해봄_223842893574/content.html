스프링 컨테이너가 대신 관리해 주는 개념입니다.

​

그런데 이 말이 이해가 잘 되시나요? 😲

저는 처음에 스프링을 배울 때 이 개념이 추상적으로 느껴져서 어려웠는데요,

그래서 오늘은 아주 쉽게 IoC와 DI 개념을 정리해 보려고 합니다. 예제와 함께요.

​

1. 기존 자바에서 객체를 생성하고 관리하는 방식

스프링을 사용하기 전, 일반적인 자바 개발에서는 객체를 개발자가 직접 생성하고 관리했습니다.

​

public interface Coffee { String getName(); } public class Americano implements Coffee { public String getName() { return "Americano"; } } public class Latte implements Coffee { public String getName() { return "Latte"; } }

​

Coffee라는 인터페이스를 정의하고, 이를 구현한 Americano와 Latte 클래스가 있습니다.

별문제가 없어 보이지만, 이 방식에는 큰 한계가 있었죠.

​

public class CoffeeShop { private Coffee coffee; public CoffeeShop() { // 현재 Americano 객체만 생성 this.coffee = new Americano(); } public void serveCoffee() { System.out.println("오늘의 커피 추천: " + coffee.getName()); } }

​

이 예시에서 CoffeeShop 클래스는 Americano 객체만 직접 생성하고 있습니다.

만약 CoffeeShop에서 다른 커피인 Latte를 사용해야 한다면, 코드를 직접 수정해야 합니다.

바로 이런 경우를 CoffeeShop 클래스는 Americano와 강하게 결합되어 있다고 할 수 있는데요,

​

즉, Latte를 사용해야 할 경우, CoffeeShop 클래스를 수정해야 하는데

이는 소프트웨어 개발에서 좋은 설계 원칙인 OCP(Open/Closed Principle, 개방-폐쇄 원칙)를 위반하는 방식입니다.

OCP는 '확장에는 열려 있고, 수정에는 닫혀 있어야 한다'는 원칙이죠.

​

또 새로운 커피 종류가 추가되어도 CoffeeShop 클래스를 수정해야 하므로, 유지보수가 어려워지고 확장성이 부족합니다.

​

이 문제를 해결하기 위해 등장한 개념이 바로 IoC, 제어의 역전입니다.

​

2. IoC (제어의 역전)

스프링이 객체를 관리하는 방식

CoffeeShop 클래스는 직접 객체를 생성하는 대신,

외부에서 Coffee 객체를 주입받는 구조로 변경되어야 합니다.

이렇게 구조를 바꾸면 클래스 간 결합도가 낮아지고, 커피 타입을 변경할 때도 클래스 자체를 수정할 필요 없이 객체만 바꿔서 주입하면 되겠죠!

​

기존 자바에서는 new 키워드를 사용해 개발자가 직접 객체를 생성하고 관리해야했지만,

스프링에서는 이 작업을 모두 스프링 컨테이너가 대신 해줍니다.

개발자는 ‘이 객체는 스프링이 관리해 줘!’라는 의미로 어노테이션 이라는 것만 붙여주면 됩니다.

​

개선된 코드 (스프링을 사용한 DI)

public interface Coffee { String getName(); } @Component public class Americano implements Coffee { public String getName() { return "Americano"; } } @Component public class Latte implements Coffee { public String getName() { return "Latte"; } }
@Component public class CoffeeShop { private Coffee coffee; // 생성자 주입을 통해 외부에서 coffee 객체를 주입받음 @Autowired public CoffeeShop(Coffee coffee) { this.coffee = coffee; } public void serveCoffee() { System.out.println("오늘의 커피 추천: " + coffee.getName()); } }

​

이제 CoffeeShop 클래스는 Coffee 인터페이스에 의존합니다.

어떤 커피 객체를 쓸지는 스프링이 알아서 주입해 주기 때문에, CoffeeShop 클래스 내부에 직접 객체를 생성할 필요가 없습니다.

커피 타입을 변경하려면 주입받는 객체만 바꾸면 됩니다. (바꾸는 방법은 다른 포스팅에서 설명합니다.)

​

이렇게 되면 새로운 커피 타입을 추가할 때도 CoffeeShop 클래스를 수정하지 않고,

새로운 커피 클래스를 추가하기만 하면 되기 때문에 OCP (개방폐쇄 원칙)을 자연스럽게 준수할 수 있습니다.

​

💡 어노테이션

이 코드에서 핵심은 @Component, @Autowired와 같은 어노테이션입니다.

이 어노테이션들은 스프링에게 "이 클래스는 스프링이 관리해야 할 객체입니다~!"라고 알려주는 신호입니다.

스프링은 이 어노테이션이 붙은 클래스들을 자동으로 생성하고, 필요한 곳에 알아서 주입해주는 거죠.

​

이렇게 등록된 클래스는 Bean이라는 이름으로 스프링 컨테이너에 등록되며,

필요한 곳에 자동으로 주입됩니다.

​

즉, 스프링이 알아서 객체를 생성하고, 생성자에 해당 객체를 주입해 주기 때문에

개발자는 객체 생성 방식에 대해 고민할 필요 없이, 필요한 객체를 주입받아 사용하기만 하면 됩니다.

이 구조 덕분에 코드도 더 깔끔해지고, 클래스 간 결합도는 느슨해져서 유지보수가 쉬워집니다.

​

​

기존 방식

3. 의존성 주입 DI

이 의존성 주입 방법으로는 아래 네 가지가 있습니다.

​

  1. @Autowired (필드 주입)

  2. Setter 주입

  3. 생성자 주입 (스프링에서 권장하는 방식)

  4. 매개변수 주입

​

이 중에서 가장 권장되는 방법은 3번, 생성자 주입입니다.

@Controller public class TestController { private final TestService testService; public TestController(TestService testService) { this.testService = testService; } }

​

의존성 주입 방법에 대한 자세한 글을 이 링크를 참고해 주세요!

의존성 주입 3가지 방법 (생성자 주입, Field 주입, Setter 주입, @RequiredArgsConstructor과 @Autowired 차이점)

순환 참조 에러 덕분에 자주쓰는 @Autowired 어노테이션과 @RequiredArgsConstructor에 대해 찾아보고 ...

blog.naver.com

​

이 '생성자 주입' 방법이 가장 권장되는 이유는

​

1) 객체가 생성될 때 반드시 의존성이 주입되므로 null을 걱정할 필요가 없고 (불변성 보장)

2) 테스트 코드 작성이 간편해져서 유연하게 Mock 객체를 넣을 수 있으며,

3) final 키워드를 사용하여 의존성이 반드시 주입되도록 강제되기 때문이이에요.

이게 무슨 뜻이냐면, final 키워드를 통해 해당 의존성을 변하지 않는 필수 값으로 간주하게 만들 수 있다는 겁니다.

​

final이 붙어 있으면 나중에 set 하거나 바꿀 수 없기 때문에, 해당 객체는 '절대적으로 필요한 값'이라는 걸 코드로 보장할 수 있다는 뜻이에요. 결과적으로 개발자가 실수로 빼먹거나 변경하지 못하도록 막아줍니다.

​

한마디로 final이 붙어 있으면 나중에 set 하거나 바꿀 수 없기 때문에,

해당 객체는 '절대적으로 필요한 값'이라는 걸 코드로 보장할 수 있다는 뜻이에요.

결과적으로 개발자가 실수로 빼먹거나 변경하지 못하도록 막아줍니다.

안정적인 객체 상태를 유지하는 데 굉장히 큰 도움이 되죠.

4. 확인

4-1. Postman이나 브라우저에서 실행해 보기

가장 간단한 방법은 실제로 애플리케이션을 실행해보고, Postman이나 브라우저를 통해 테스트하는 것입니다.

예를 들어, TestController에 매핑된 주소가 /coffee라면 http://localhost:8080/coffee 와 같이 직접 접속해서, 응답으로 어떤 커피가 출력되는지 확인해볼 수 있습니다.

이렇게 요청을 보내보면, 우리가 직접 new 하지 않았던 Coffee 객체가 자동으로 주입되어 동작하는 걸 확인할 수 있죠.

​

4-2. 스프링이 객체를 자동으로 생성하는 과정 디버깅

조금 더 깊게 확인하고 싶다면,

TestController 클래스의 생성자에 중단점을 걸고 디버깅을 실행해 보세요.

생성자에 진입하는 순간, 스프링이 어떻게 testService 같은 의존 객체를 주입해주는지 과정을 따라가볼 수 있습니다!

이 과정에서 우리가 명시적으로 new TestService() 같은 객체 생성을 하지 않았음에도, 스프링이 미리 생성해 둔 Bean을 주입해주는 것을 확인할 수 있습니다.

즉, 객체 생성과 주입을 개발자가 하지 않아도, 스프링이 알아서 처리해주고 있는 셈이죠.

​

4-3. Bean 조회해 보기

마지막으로 스프링이 생성한 Bean 목록을 직접 확인하는 방법도 있습니다.

아래 코드를 실행하면, 스프링 컨테이너에서 관리하는 모든 Bean의 이름이 출력될 것입니다.

import org.springframework.context.ApplicationContext; import org.springframework.context.annotation.AnnotationConfigApplicationContext; import java.util.Arrays; public class BeanCheck { public static void main(String[] args) { ApplicationContext context = new AnnotationConfigApplicationContext(DemoApplication.class); String[] beanNames = context.getBeanDefinitionNames(); Arrays.stream(beanNames).forEach(System.out::println); } }

​

코드를 실행하면, 스프링 컨테이너가 어떤 객체들을 관리하고 있는지 리스트로 쭉 출력됩니다.

이걸 보면 “아, 진짜 스프링이 다 관리해주는구나!” 라는 게 실감 날 거예요.

스프링이 IoC 컨테이너에서 객체를 어떻게 생성하고 관리하는지 쉽게 이해되셨나요?

📝 IoC와 DI 핵심 요약

IoC (Inversion of Control)

객체의 생성과 관리를 개발자가 아니라 스프링 컨테이너가 담당

​

DI (Dependency Injection)

객체 간 의존 관계를 스프링이 자동으로 주입
