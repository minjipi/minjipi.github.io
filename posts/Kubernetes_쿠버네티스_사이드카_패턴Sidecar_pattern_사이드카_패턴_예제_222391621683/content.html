사이드카 패턴(Sidecar) 이란?

사이드카 패턴은 원래 사용하려고 했던 기본 컨테이너의 기능을 확장하거나 강화하는 용도의 컨테이너를 추가하는 패턴이다.

즉, 기본 컨테이너와 독립적으로 동작하는 별도의 컨테이너를 붙이는 패턴이기 때문에 어플리케이션 컨테이너의 변경이나 수정 없이 독립적으로 동작하는 컨테이너를 붙였다 뗐다 할 수 있다.

옆에 붙은 사이드카를 붙이든 떼어내든 상관 없이 오토바이는 동일하게 운전할 수 있는 것처럼.

사이드카 패턴(Sidecar) 예제

사이드카 패턴을 이용해서 다음 그림과 같은 Pod를 구성해보자.

​

하나는 서비스 하는 컨테이너, 하나는 받아오는 컨테이너.

​

우선 사이드카 디렉토리를 생성한다.

mkdir /sidecar cd /sidecar

/sidecar 안에 아래의 [ Dockerfile ], [ contents-pull.sh ] 두 파일을 생성한다.

​

​

[ Dockerfile ]

FROM ubuntu:16.04 RUN apt-get update && apt-get install -y git ADD ./contents-pull.sh /contents-pull.sh RUN chmod 755 /contents-pull.sh WORKDIR / cmd /contents-pull.sh

​

[ contents-pull.sh ]

#!/bin/bash if [ -z $CONTENTS_SOURCE_URL ]; then exit 1 fi git clone $CONTENTS_SOURCE_URL /data cd /data while true do date sleep 60 git pull done

​

​

도커 허브에 sidecar 레포지터리 만들고,

​

​

이미지 생성

docker build --tag ghdalswl77/sidecar:1.0 ./

​

환경변수 옵션 지정하는 옵션줘서, 컨테이너 생성

docker run -e CONTENTS_SOURCE_URL=https://github.com/takara9/web-contents --name test -d ghdalswl77/sidecar:1.0

​

레포지터리에 이미지 업로드

docker push ghdalswl77/sidecar:1.0

​

[sidecar.yaml]

apiVersion: v1 kind: Pod metadata: name: sidecar spec: containers: - name: nginx image: nginx volumeMounts: - mountPath: /usr/share/nginx/html name: contents-vol - name: puller image: [도커허브ID]/[레포지터리]:[버전] env: - name: CONTENTS_SOURCE_URL value: "https://github.com/takara9/web-contents" #그냥 간단한 html 페이지! volumeMounts: - mountPath: /data name: contents-vol volumes: - name: contents-vol emptyDir: {}

이 파일을 자세히보자.

우선, 두 개의 컨테이너가 '하나의 볼륨'을 사용하게 해주었다.

​

mountPath 에 각각 위와 같이 작성하여

하나의 하드디스크를 같이 쓰게 해 준다.

​

그리고 puller 컨테이너는 컨텐츠를 1분에 한 번씩 받아오고,

nginx 컨테이너는 컨텐츠를 사용자들이 요청하면 전송해준다.

​

​

위 파일들로- pod 생성

kubectl apply -f [파일명]

​

생성한 pod IP 확인:

kubectl get pods -o wide

​

잘 됐나~ 접속해본다.

curl http://사이드카-IP

사이드카 패턴(Sidecar) ++

사이드카 패턴을 사용하기 좋은 샘플들은 다음과 같겠다.

​

- 보안을 위해 사이드카로 NGINX reverse proxy 등을 붙여서 HTTPS 통신을 한다.

- 성능을 위해 사이드카로 NGINX content cache 등을 붙인다.

- 컨테이너 외부로 로그를 모으기 위해 logstash, fluentd 등을 붙인다. (centralized logging)

​

모두 어플리케이션 변경 없이, 사이드카를 붙였다 떼거나 교체하기 쉬운 구성 방식이다.

상황에 따라 사이드카를 다른 것으로 변경하거나 버전업 등이 필요할 수도 있다.

예를 들어, content cache를 NGINX 대신 Apache HTTP server로 교체하는 등의 작업을 해도 어플리케이션은 영향을 받지 않는다.

​

재사용성이 높아지고, 생산성이 높아지는 등, 이런저런 장점이 많겠지만

어플리케이션이 너무 작을 경우엔 배보다 배꼽이 커지는 상황에 주의해서 사용해야할 것 같다!
