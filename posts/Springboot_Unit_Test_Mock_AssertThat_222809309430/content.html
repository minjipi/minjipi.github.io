Springboot Unit Test 공부하며 정리

Mock, AssertThat

1.

build.gradle에 다음을 추가

testImplementation 'org.springframework.boot:spring-boot-starter-test'

​

@SpringBootTest 어노테이션

@SpringBootTest가 하는 역할: @SpringBootApplication을 찾아서 테스트를 위한 빈들을 다 생성한다.

그리고 @MockBean으로 정의된 빈을 찾아서 교체한다.

​

@MockBean

컨트롤러만 테스트하고 싶을 경우 서비스 객체를 MockBean으로 만들어서 사용할 수 있다.

ApplicationContext에 들어있는 빈을 Mock으로 만든 객체로 교체함

모든 @Test 마다 자동으로 리셋. 직접 리셋을 관리 하지 않아도 된다.

​

2.

테스트 코드를 작성할 때, 크게 두 부분으로 나누어서 생각할 수 있다.

​

1. 실행 단계

2. 검증 단계

​

실행 단계에서는 사용자 입력을 모방하여 테스트할 로직을 실행시키고

검증 단계에서는 실행 단계가 예상한 대로 동작하는지 검증한다.

​

JUnit에서 이 검증 단계를 도와주는 메소드가 바로 org.junit.Assert.assertThat다.

​

​

사용하려면 다음을 import

import static org.assertj.core.api.Assertions.assertThat;

​

테스트 케이스에서 사용 예시 >

@SpringBootTest public class OrderServiceTest { @Nested class Success { @Test @DisplayName("주문 성공") public void 주문번호_성공() { Order order = Order.builder() .orderNum("1COFFEE9MILK2") .productName("어리틀바닐라목걸이") .build(); System.out.println(order.getOrderNum().length()); assertThat(order.getOrderNum().length()).isEqualTo(12); } }

​

assertEquals와 비교해서 가독성, Failure 메세지, 타입 안정성 등 장점들이 있다는데..

자세한 부분은 다음 링크를 참고 : https://jongmin92.github.io/2020/03/31/Java/use-assertthat/

, https://johngrib.github.io/wiki/junit5-nested/

TDD는 처음인데, 오.. 느끼는게 많았다.

좋은 점과 나쁜점에 대해 기록해 두려고 한다.

​

좋은점은 코드를 더 깔끔하게 짤 수 있다는 점이다.

미리 어떻게 만들 것인지 테스트를 해보고 들어가기 때문에..

어떤 분이 외국 개발자 분들이랑 일할때 초반 몇 달을 변수명 정하고 기능 정리만 정리했다던데, 처음엔 의아했지만 결국 개발 진행 속도가 훨씬 빠르고 깔끔했다는 이야기가 생각났다.

테스트를 먼저 해봄으로써 전체적으로 구조 파악이 되고, 결국 코드가 더 객체 지향적인 코드가 되는 것 같다.

​

아쉬운 점은... 역시 아직 내 실력이 부족하다는 것. 그래서 테스트 코드 작성에 시간이 더 걸린다는 것..ㅋㅋㅋ

이건 TDD의 단점이라기 보단 내 단점이지만, 어쨌든 코드 양이 늘어나긴 하니 그만큼 시간 비용이 더 들 수 있다는 점?
